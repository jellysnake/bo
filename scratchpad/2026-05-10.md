# 2026-05-10 — Priority Ranking

Top 3 implementation priorities for the next sessions, ordered by impact against: user value at release, alpha testability, and core product loop stickiness.

## 1. `bo query` with BYOK LLM synthesis and deterministic fallback

**What:** A single `bo query <question>` command that retrieves relevant leaves via the existing search infrastructure, then synthesizes an answer with citations. When no API key is present, falls back to presenting retrieved sources with citations (deterministic mode).

**Why this is #1:**
- Closes the core loop (`collect → compile → query`). Without it, bo is a static file generator.
- The "magic moment" — a user asks their knowledge base a question and gets an answer citing their collected sources.
- Search already satisfies the "deterministic before LLM-dependent" sequencing principle. A separate deterministic-only query step is redundant with search and would be discarded once synthesis ships.
- Provider infrastructure exists from compile; incremental rather than greenfield.
- Immediately testable in alpha against any dogfood tree.

**Design notes:** Graceful degradation (no key → citation list, key present → synthesized answer) in one command. Retrieval layer is existing `bo search` internals. New work is context assembly, synthesis prompt, citation extraction, and output formatting.

## 2. `--json` output flag on all commands

**What:** A `--json` flag on collect, compile, list, show, search, and query that produces structured JSON output suitable for programmatic consumption.

**Why this is #2:**
- Bo's architectural stance is "external agents compose bo commands" and "MCP server is a natural extension." Machine-parseable output is table stakes for the product thesis.
- Every command shipped without `--json` is retrofit debt. Doing it now while command surfaces are fresh is cheaper than bolting it on later.
- Low complexity, parallelizable with other work.
- Schemas defined now become the future MCP contract.
- Testable immediately — agents in alpha sessions can consume structured output.

**Design notes:** Schema design is the real architecture here. Keep schemas minimal and stable. Defer formal MCP server until schemas prove out in practice.

## 3. Compile summary-based fallback for context overflow

**What:** When collected leaf content exceeds the compile model's context window, substitute full leaf bodies with their pre-generated summaries (already stored in frontmatter at collect time).

**Why this is #3:**
- Prevents compile failure at moderate scale without requiring users to know which model has which context limit.
- A workaround exists today (manually set `compile_model` to a larger-context model), so this is defensive rather than urgent.
- Summary generation is already implemented; this is consumption-side wiring only.
- Unblocks the collect→compile path at realistic scale for release users who won't read workaround docs.

**Design notes:** Needs token estimation heuristic, fallback trigger threshold, and a decision on partial overflow (some leaves full, some summarized, vs. all-or-nothing switch). Medium architectural design load.

## What was dropped

- **Deterministic `bo query` MVP as a separate feature** — redundant with `bo search`. The sequencing principle is satisfied by search existing. Merged into #1 as the fallback mode.
- **Compile fallback urgency** — demoted from #2 to #3 because the manual workaround (bigger model) covers alpha usage. Becomes important at beta/release when users won't know the workaround.
