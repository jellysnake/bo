# 2026-05-08

## Architecture mental model

Reorganising src/ into this structure now. The codebase is small enough that the
move is mechanical (mkdir, mv, update use paths) and doing it now means every new
file lands in the right place from the start rather than accumulating flat debt.

### Target shape

```
src/
├── main.rs              # CLI dispatch only
├── lib.rs               # public API surface, re-exports
│
├── domain/              # pure types, invariants, zero deps
│   ├── tree.rs
│   ├── branch.rs
│   ├── leaf.rs
│   ├── frontmatter.rs
│   ├── slug.rs
│   └── index.rs
│
├── cli/                 # orchestration pipelines (collect, compile, list, quality)
│   ├── collect.rs
│   ├── compile.rs
│   ├── list.rs
│   └── quality.rs
│
├── engine/              # shared capabilities cli/ depends on
│   ├── fetch.rs
│   ├── extract.rs
│   ├── agent.rs
│   └── config.rs
│
└── adapters/            # external world protocols
    └── youtube/
```

### Dependency graph

```
                    ┌─────────┐
                    │  main   │  CLI surface
                    └────┬────┘
                         │ dispatches to
              ┌──────────┼──────────┐
              ▼          ▼          ▼
         ┌────────┐ ┌────────┐ ┌──────┐
         │collect │ │compile │ │ list │   cli/
         └───┬────┘ └───┬────┘ └──┬───┘
             │          │         │
             ▼          ▼         ▼
        ┌─────────────────────────────┐
        │  engine (fetch, extract,    │   engine/
        │  agent, config)             │
        └──────────┬──────────────────┘
                   │
          ┌────────┴────────┐
          │ reaches outside │
          ▼                 ▼
┌──────────────────┐   ┌─────────────────────────┐
│  adapters/       │   │  domain (tree, branch,   │
│  (youtube, ...)  │   │  leaf, frontmatter,      │
└──────────────────┘   │  slug, index)            │
                       └─────────────────────────┘
                         ▲ ▲ ▲    pure types,
                         │ │ │    zero dependencies
                         │ │ │
              all layers construct/read domain
```

### Why this shape

- **domain/** is the gravity well. Every layer depends on it; it depends on
  nothing. These aren't ORM "models" — they encode the tree/branch/leaf metaphor
  that IS the product. Pure data + invariants + transformations.

- **engine/** does singular jobs: fetch HTML, extract content, run the agent loop.
  It's the layer that crosses the network boundary by consuming adapters.

- **cli/** orchestrates engine modules into user-facing pipelines. One file per
  subcommand. Thin glue, no business logic of its own.

- **adapters/** encapsulates protocol-specific knowledge (innertube, transcript
  parsing). Consumed only by engine. New source types (RSS, Pocket export, etc.)
  land here without touching anything above.

- Dependencies flow strictly downward. No upward or lateral arrows between layers.
  domain never knows adapters exist. cli never imports from adapters directly.
