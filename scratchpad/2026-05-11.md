# 2026-05-11 — Query V1 hardening take

After completing `specs/chore/query-v1-operational-hardening`, `bo query` feels materially better operationally. It is no longer just a happy-path prototype; it is now reasonable to dogfood as V1.

## What improved

- LLM calls are bounded: query, compile, and summary now use explicit timeout/retry policies.
- Provider failures are explicit: timeout, retry exhaustion, truncation, and content filter outcomes surface as errors instead of hanging, silently degrading, or parsing partial output.
- Query context is model-aware for known OpenAI models and fails clearly for unknown model context windows.
- Citations are stronger: final Sources/JSON citations now reflect valid prose wikilinks plus structured `cited_slugs`, deduped in stable order.
- Invalid wikilink cleanup is deterministic and no longer requires a direct `regex` dependency.
- Future mdBook/Rust Book collections should no longer write `Keyboard shortcuts` as the title when a content heading like `Understanding Ownership` is available.
- Summary generation no longer falls back in gray-zone attempted-provider failures; collection aborts before partial writes.
- Live smoke on rebuilt `tmp-tree` from `corpora/default/urls.txt` passed, and query cited `[[understanding-ownership]] — Understanding Ownership` plus `[[fearless-concurrency]] — Fearless Concurrency`.

Key catch: V1 no-results behavior only triggers when lexical retrieval finds zero hits. With OR substring retrieval, unrelated natural-language queries often retrieve generic-term matches (`work`, `structure`, `mechanics`, etc.). The model usually refuses or says sources do not cover the topic, but it can still answer from outside knowledge with zero citations. PMNS query demonstrated this failure.

Follow-up candidate: if synthesis yields zero valid citations, treat the result as not answered from the tree instead of successful query output. Also consider generic-term filtering or a retrieval relevance floor.

## Remaining caveats

1. Retrieval is still naive/full-scan. Fine at current dogfood size, still the next scaling concern for hundreds/thousands of leaves.
2. Query recall is lexical. V1 can miss docs when user wording does not overlap source wording. That is expected until semantic or agentic V2 retrieval.
3. Context budgeting is a static model table. Good for current OpenAI models, intentionally rejects unknown/custom/local models.
4. `.env` ergonomics remain annoying: the local `.env` uses shell command substitution, which `dotenvy` does not execute. Manual smoke tests need `source .env && bo ...`, or `.env` should become literal dotenv syntax.
5. Title heuristic is intentionally narrow. It fixed the observed `Keyboard shortcuts` issue without broad guessing; future title fixes should be fixture-driven.

## Take

V1 query is now operationally respectable. It is safe to keep dogfooding on the current corpus and gather retrieval-quality evidence before moving to V2. The next small follow-up should probably be `bo config set query_model ...` or deterministic tree health/status tooling, not more query internals yet.
