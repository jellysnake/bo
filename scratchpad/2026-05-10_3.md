# 2026-05-10 — Query V1 Implementation Critiques & Follow-up

## Module size

`cli/query.rs` is ~480 lines handling 6 responsibilities: term extraction, leaf retrieval, context assembly, synthesis, citation validation, and output formatting. As V2 (agentic retrieval per ADR-003) arrives, this becomes a refactoring tax. Recommendation: split into `cli/query/` directory with sub-modules (`extract.rs`, `retrieve.rs`, `assemble.rs`, `synthesize.rs`) before adding V2 complexity.

## Retrieval performance

Currently reads ALL leaf files from disk on every query (O(n) full file reads). At 54 leaves this completes in <50ms. At 300+ leaves this will be noticeable. The index (`index.jsonl`) already has titles — a two-phase approach (score against index metadata first, then read full files only for top-k candidates) would reduce I/O significantly. No warning or graceful degradation currently exists.

## Upstream data quality surfaces through query

The dogfood test showed 'Keyboard shortcuts' citations — Rust Book pages have titles extracted from UI chrome rather than page content. Query makes this more visible/painful than `bo list` because citations appear in the synthesized answer. The 'Improve extracted titles when UI chrome pollutes document metadata' item in `features.md` is now higher priority post-query. Query retrieval scores against title+summary+body, so garbage titles also dilute ranking accuracy.

## No timeout or retry on LLM calls

The synthesis call (via `tokio block_on`) has no timeout. A network hiccup or slow response hangs the command indefinitely. This is consistent with compile (same pattern) but both should eventually get a configurable timeout. Also no retry on transient 429/5xx errors.

## regex dependency for one operation

Added the `regex` crate (~1MB compile cost) for a single `replace_all` call stripping invalid `[[slug]]` wikilinks from answer prose. A 15-line character-by-character parser for the `[[...]]` pattern would have been sufficient. Not urgent but worth noting if dependency count becomes a concern.

## Token budget is a heuristic with no model awareness

The 60k word (~80k token) budget cap is hardcoded. If a user sets `query_model` to a model with 8k context, the assembled prompt will silently overflow. No validation against known model limits. V1-acceptable but a footgun for users with smaller models.

## Next steps for query

1. Upstream: fix title extraction quality (`features.md` item) — highest impact on query UX
2. Consider the two-phase retrieval optimization if trees grow past ~200 leaves
3. V2 agentic retrieval (ADR-003) — trigger is when dogfood shows search-based retrieval consistently missing relevant content that a human would find by browsing
4. Gemini provider support (stretch goal mentioned during spec discussion)
5. Cross-query context/sessions (ADR-003 open direction) — deferred until V2
