## Collection issue analysis

- Core issue: `collect` treats `HTTP 2xx + HTML content-type + trafilatura body over 50 chars` as success. That is too weak; boilerplate, redirect stubs, JS-disabled shells, and app footers can all satisfy it.
- Fetch layer only follows HTTP redirects. It does not follow deterministic HTML/JS redirect stubs such as Rust blog's `window.location.replace(...)` / `meta refresh` page.
- Extraction layer trusts generic trafilatura output as the document. It does not classify low-value output or fall back to structured metadata/app data.
- Source-specific observations:
  - Medium: real fetch/access failure; Cloudflare challenge returns `403`, so current failure is expected unless using a different fetch strategy or explicit unsupported/blocked classification.
  - X/Twitter: fetch can return a large HTML app/error shell; visible extracted text is the JS-disabled page, not tweet content. Needs source adapter or explicit unsupported/blocked detection.
  - OpenReview: useful title/abstract are present in meta/citation tags, but visible extraction returns shell/footer. Needs metadata/API fallback or adapter.
  - Rust blog: deterministic redirect stub; should be resolved before extraction.
  - Rust book/mdBook: HTML has correct `<title>` and `<main><h1>`, but extractor chooses `Keyboard shortcuts` from UI chrome. Needs DOM cleanup/title heuristic for docs pages.
