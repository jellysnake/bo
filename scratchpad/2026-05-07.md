## Current project assessment

The project is now a credible dogfood alpha, not yet something to present as stable public software.

Comfortable open-source framing today:

> experimental local markdown knowledge-tree tool; APIs/format may change

Not yet comfortable framing:

> stable/reliable general-purpose knowledge management tool

The key missing loop:

```text
collect → compile → query/read/navigate → refine tree → repeat
```

Current state is closer to:

```text
collect → compile → files exist
```

The core storage philosophy still looks right:

- Markdown tree is user-owned and inspectable.
- Any database/index/snapshot layer should be acceleration/history/state safety, not primary source of truth.

## Synthesized roadmap themes

### 1. Query path

`bo query` is the main missing product loop.

MVP should likely be lexical/deterministic first:

```bash
bo query "what are the strongest arguments against LLM scaling?"
```

Initial implementation shape:

- search local leaves/branches lexically
- answer from top matches
- cite files/URLs
- do not mutate tree

Likely staged path:

1. lexical search/query MVP
2. provider API-key LLM answer synthesis
3. clean entrypoint for coding clients / local agent clients
4. local LLM server support:
   - llama.cpp
   - vLLM
   - Ollama
   - OpenAI-compatible local endpoints

Provider priority for bring-your-own-key release:

1. OpenAI
2. Anthropic
3. OpenRouter
4. Gemini / Google
5. Copilot / Azure
6. Bedrock / AWS

BYOK is the right first release model. It avoids backend/SaaS complexity and gets the tool into users' hands.

### 2. Tree inspection/navigation

Useful deterministic commands before or alongside query:

```bash
bo list
bo show <slug>
bo search <term>
bo status
```

Purpose:

- user can see what exists without manually browsing files
- agents can discover tree state with small deterministic calls
- low risk, high trust

Possible examples:

```bash
bo list --recent
bo list --source youtube
bo show stop-slicing-the-elephant...
bo search "scaling laws"
bo status
```

### 3. Tree health / repair

Need commands for users and machines to inspect broken tree state.

Possible command names:

```bash
bo scan
bo survey
bo doctor
bo prune <slug-or-id>
bo rebuild-index
```

Preferred framing from discussion:

- `bo survey` / `bo scan`: detect issues, no mutation
- `bo prune <slug-or-id>`: remove dead leaf/branch intentionally
- `bo rebuild-index`: rebuild derived index from markdown frontmatter

Checks to include:

- broken branch references
- bad YAML frontmatter
- missing index entries
- index entries pointing to missing files
- empty files
- duplicate URLs
- orphan markdown files
- suspicious path traversal entries
- dead leaves/branches
- missing/invalid branch metadata

Lower effort MVP:

- scan/survey reports issues
- user/agent fixes one issue at a time
- destructive operations explicit, never hidden

### 4. State snapshots / history / blast-radius reduction

Need safety around scenarios like:

- user breaks content manually
- LLM breaks content
- index corruption
- compile writes bad branches
- partial operation failure

Potential MVPs:

- content hash manifest snapshots
- JSON snapshot manifests
- git-backed snapshots
- git-like internal object store
- SQLite metadata table later if query/indexing needs it

Important constraint:

- Markdown tree should remain authoritative.
- DB/snapshot layer should be rebuildable where possible.
- Consider storing snapshots under user-local `~/.bo` to reduce blast radius of tree operations.

Possible commands:

```bash
bo snapshot
bo snapshots
bo diff <snapshot-a> <snapshot-b>
bo restore <snapshot>
```

Open design question:

- Should snapshots live inside the tree for portability, or in `~/.bo` for safer isolation?
- Hybrid is possible: local metadata store + optional exportable snapshot manifest.

### 5. Failure/retry ledger

Less urgent than query/inspection/health.

Still useful later:

```bash
bo rejected
bo retry
bo retry-rejected
```

Manual retry is cheap today, so this is QoL rather than core product readiness.

Possible storage:

```text
.bo/events.jsonl
```

Events:

- collected
- rejected
- duplicate
- fetch failed
- compile started/completed
- query run maybe not by default

### 6. Compile hardening and agent lifecycle

This is likely the largest reliability area.

Mental model for compile:

1. Give the LLM/agent composable deterministic tools.
2. Harness should focus model attention; avoid broad unrestricted file access where deterministic tools can narrow context.
3. Agent traverses a contained snapshot of the tree before making edits.
4. Agent plans writes at a high level.
5. Agent identifies patterns/branches using progressive disclosure:
   - read index first
   - then frontmatter
   - then partial/full content only for relevant files
6. Use cheap/focused subagents for scoped tasks:
   - read/summarize one file
   - inspect a cluster
   - draft a branch
   - verify a proposed edit
7. Final gate before writes:
   - sanity-check planned writes
   - validate branch/frontmatter structure
   - ideally use a small/specialized model or deterministic validator as review gate
8. Only then apply edits.

Release-oriented compile needs:

```bash
bo compile --dry-run
bo compile --snapshot
bo compile --model ...
```

Also needed:

- clearer provider/model config
- branch write validation
- recovery after failed compile
- diff preview before writes
- no invented leaf refs
- deterministic guards around generated outputs

## Current recommendation

Next best feature sequence:

1. `bo status` or `bo list` — quick deterministic user confidence.
2. `bo survey` / scan — tree health and repair visibility.
3. `bo search` — deterministic retrieval groundwork.
4. `bo query` lexical MVP — completes the core loop.
5. compile safety/snapshot work — protects generated writes.

Avoid jumping straight to a DB-backed query system until the deterministic inspection/search layer exists.
