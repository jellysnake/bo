# Data Model: Leaf Summary

## Modified entity: Leaf frontmatter

### Before

```yaml
---
title: "Document Title"
url: https://example.com/article
collected_at: 2026-05-09T22:00:00Z
updated_at: 2026-05-09T22:00:00Z
---
```

### After

```yaml
---
title: "Document Title"
url: https://example.com/article
collected_at: 2026-05-09T22:00:00Z
updated_at: 2026-05-09T22:00:00Z
summary: |
  A ~200-word prose paragraph describing the document content, optimized
  for retrieval triage. Generated by LLM when available, otherwise the
  first ~200 words of body content.
---
```

## Field specification

| Field | Type | Required | Source |
|-------|------|----------|--------|
| `summary` | string (YAML literal block scalar) | Yes (on new collects) | LLM or deterministic fallback |

## Storage format

Multi-line summaries use YAML literal block scalar (`|`) to:
- Avoid escaping issues with quotes, colons, special characters
- Preserve readability in the raw markdown file
- Round-trip cleanly through `frontmatter::parse` / `frontmatter::patch_fields`

Single-line summaries (short fallbacks) can use plain scalar.

## Backward compatibility

- Existing leaves without `summary:` field continue to work — all consumers treat it as optional.
- `frontmatter::parse` returns a `Mapping` — absent keys are simply `None` on lookup.
- No migration needed. Summaries appear only on newly collected leaves.

## Constants

```rust
const SUMMARY_TARGET_WORDS: usize = 200;   // target summary length
const SUMMARY_INPUT_MAX_WORDS: usize = 4000; // body truncation before LLM call
```
