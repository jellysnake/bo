# v0.0.2 Milestone: Compile Grows Up

**Status:** Planning  
**Target:** The compile loop becomes production-grade for growing trees.

## Narrative

v0.0.1 proved the collect→compile→query loop works. v0.0.2 makes compile the intelligent part — incremental, safe, aware of its own state. The tree compounds knowledge sustainably rather than requiring expensive full recompilation.

This is the first release where a visitor sees "this *thinks* about my collection" rather than "this is a script I could write in a weekend."

---

## Implementation Backlog

### 1. `bo status`

**Goal:** Users and agents can see tree health and compile readiness at a glance.

- [x] Leaf count and branch count from index/filesystem
- [x] Last compile timestamp (persist in config or manifest)
- [x] List of leaves added since last compile (uncompiled leaves)
- [x] Tree size in bytes and estimated tokens
- [x] Index health: orphan entries, missing files
- [x] Human-formatted output (table/summary)
- [x] `--json` output
- [x] Unit tests: status against known tree fixtures
- [x] Integration test: collect → status shows uncompiled leaf

**Done when:** `bo status` after collecting 3 new leaves shows them as uncompiled, displays tree size, and reports last compile time.

---

### 2. Compile validation gate

**Goal:** Compile never writes malformed branches. Bad LLM output is caught and rejected before any file mutation.

- [x] Validate structured LLM response shape (expected fields present, correct types)
- [x] Verify all referenced leaf slugs exist in the tree
- [x] Reject empty branch bodies or invented file paths
- [x] Sanity check: diff size is not absurdly large relative to input
- [x] On validation failure: surface errors, write nothing, exit non-zero
- [x] Unit tests: valid response passes, various malformed responses rejected
- [x] Integration test: mock LLM returns bad output → no files written, clear error

**Done when:** A deliberately malformed compile response (missing fields, phantom leaf references, empty bodies) is rejected with no partial writes.

---

### 3. Denormalized manifest and write-intent recovery

**Goal:** Single source of truth for tree topology with crash-safe mutations. Foundation for incremental compile, dry-run, and future DB migration.

See [ADR-004](../adrs/004.md) for full design rationale and schema.

- [ ] Define `Manifest`, `LeafRecord`, `BranchRecord` types in `src/domain/manifest.rs`
- [ ] Define `PendingOperation` types in `src/engine/pending.rs`
- [ ] `manifest::read()` / `manifest::write()` with atomic rewrite (write tmp + rename)
- [ ] Migration: detect `index.jsonl` without `manifest.json` → auto-migrate on first run
- [ ] `bo seed` writes initial `manifest.json` (empty leaves/branches)
- [ ] `bo collect` reads/writes manifest instead of appending to `index.jsonl`
- [ ] `bo compile` reads/writes manifest instead of scanning `branches/` + `state.json`
- [ ] `bo status` reads manifest directly (no filesystem scan for topology)
- [ ] `bo raze` deletes manifest as part of tree teardown
- [ ] Write-intent: mutating commands write `pending.json` before mutations, delete after
- [ ] Recovery: detect stale `pending.json` on startup, warn and resolve
- [ ] Remove `index.jsonl` and `state.json` code paths (post-migration)
- [ ] Unit tests: manifest CRUD, migration from index.jsonl, pending write/recovery
- [ ] Integration tests: collect → crash simulation → recovery → consistent state

**Done when:** All tree state lives in `manifest.json`. A killed process mid-compile leaves a recoverable `pending.json`. `index.jsonl` and `state.json` are gone.

---

### 4. Incremental compile

**Goal:** Default compile processes only new leaves and fits them to existing branch structure. Full recompile is opt-in.

- [ ] On `bo compile`: identify leaves where `branch_assignments` is empty (uncompiled)
- [ ] If no uncompiled leaves and no stale branches: report "nothing to compile", exit 0
- [ ] LLM prompt receives: existing branch structure (from manifest) + new leaf summaries
- [ ] LLM outputs: assignments of new leaves to existing branches, and/or new branch definitions
- [ ] Existing branch prose is untouched (new leaves are appended/integrated, not full rewrite)
- [ ] `--all` flag: recompile full corpus regardless of assignment state (current behavior)
- [ ] Detect deleted leaves: if a leaf in `branch.leaf_members` no longer exists in manifest, flag `stale: true`
- [ ] `--rebuild-stale` flag: recompile only stale branches from their remaining leaves
- [ ] Update `tree.last_compiled_at` on successful compile
- [ ] `--dry-run` flag: build pending.json, print proposed changes, exit without writing
- [ ] `compile_model` config key threaded through (may need different model from query)
- [ ] Unit tests: incremental selection logic, stale detection, dry-run output
- [ ] Integration tests: collect → compile → collect more → compile again (only new processed)

**Done when:** A 50+ leaf tree with 3 new additions compiles in seconds (touching only new leaves), a manually deleted leaf causes its branch to be flagged stale, and `--dry-run` previews changes without writing.

---

### 5. CLI rename and command rework

**Goal:** Command names reflect the botanical metaphor and the library navigation protocol. Merge redundant commands.

- [ ] `collect` → `feed` (nourish the tree with sources)
- [ ] `compile` → `grow` (the tree grows branches from leaves)
- [ ] `search` removed — merged into `list [<terms>]`
- [ ] `list` gains hierarchy awareness: default shows branch structure, `--branches`/`--leaves` for filtering, positional args for text search
- [ ] `show` defaults to card view (frontmatter/summary), `--full` for content
- [ ] `show` on a branch displays section overview + leaf list
- [ ] `show --json` omits `content` field unless `--full` is passed
- [ ] Update all help text, error messages, docs
- [ ] Alias `collect` → `feed` and `compile` → `grow` during transition (remove aliases in v0.1.0)

**Done when:** `bo feed <url>`, `bo grow`, `bo list`, `bo show <slug>` all work with new semantics. `bo search` is gone. `bo show` defaults to card view.

---

### 6. Query no-answer hardening

**Goal:** When retrieval finds irrelevant content, query reports "no answer" rather than synthesizing from noise.

- [ ] Implement retrieval relevance floor (minimum match quality before invoking synthesis)
- [ ] Generic-term filtering: very common terms that match many unrelated leaves don't trigger synthesis
- [ ] If synthesis runs but produces zero valid citations: treat as not-answered (extend existing zero-citation logic)
- [ ] Human output: clear "no relevant sources found" messaging
- [ ] JSON output: `insufficient_sources` or `low_relevance` error code
- [ ] Unit tests: below-threshold retrieval → no synthesis call
- [ ] Integration test: query unrelated topic against a themed tree → explicit not-answered

**Done when:** `bo query "quantum chromodynamics"` against a Rust programming tree exits with clear "no answer" rather than attempting synthesis against string-matched noise.

## Dependencies and sequencing

```
bo status (done) ───────┐
validation gate (done) ─┤
raze auth (done) ───────┤
                        │
                        ▼
         manifest (ADR-004) ──→ incremental compile (grow)
                                       │
                                       │  (depends on manifest for state tracking,
                                       │   write-intent for crash safety + dry-run)
                                       │
CLI rename ── independent, can land anytime (touches all commands)
query hardening ── independent, can ship anytime
```

## Success criteria

- A tree with 50+ leaves can `bo compile` in seconds (only processing new leaves), not minutes.
- `bo status` tells you at a glance what's new and uncompiled.
- Compile never writes malformed branches (validation gate catches bad LLM output).
- Query never presents hallucinated answers as grounded (zero-citation = explicit "no answer").

## Release mechanics

1. Implement scope items (one spec/PR per item).
2. Update CHANGELOG.md.
3. Bump version in Cargo.toml to `0.0.2`.
4. Merge, tag `v0.0.2`, push tag → release workflow runs.
