# v0.0.2 Milestone: Compile Grows Up

**Status:** Planning  
**Target:** The compile loop becomes production-grade for growing trees.

## Narrative

v0.0.1 proved the collect→compile→query loop works. v0.0.2 makes compile the intelligent part — incremental, safe, aware of its own state. The tree compounds knowledge sustainably rather than requiring expensive full recompilation.

This is the first release where a visitor sees "this *thinks* about my collection" rather than "this is a script I could write in a weekend."

---

## Implementation Backlog

### 1. `bo status`

**Goal:** Users and agents can see tree health and compile readiness at a glance.

- [ ] Leaf count and branch count from index/filesystem
- [ ] Last compile timestamp (persist in config or manifest)
- [ ] List of leaves added since last compile (uncompiled leaves)
- [ ] Tree size in bytes and estimated tokens
- [ ] Index health: orphan entries, missing files
- [ ] Human-formatted output (table/summary)
- [ ] `--json` output
- [ ] Unit tests: status against known tree fixtures
- [ ] Integration test: collect → status shows uncompiled leaf

**Done when:** `bo status` after collecting 3 new leaves shows them as uncompiled, displays tree size, and reports last compile time.

---

### 2. Compile validation gate

**Goal:** Compile never writes malformed branches. Bad LLM output is caught and rejected before any file mutation.

- [ ] Validate structured LLM response shape (expected fields present, correct types)
- [ ] Verify all referenced leaf slugs exist in the tree
- [ ] Reject empty branch bodies or invented file paths
- [ ] Sanity check: diff size is not absurdly large relative to input
- [ ] On validation failure: surface errors, write nothing, exit non-zero
- [ ] Unit tests: valid response passes, various malformed responses rejected
- [ ] Integration test: mock LLM returns bad output → no files written, clear error

**Done when:** A deliberately malformed compile response (missing fields, phantom leaf references, empty bodies) is rejected with no partial writes.

---

### 3. Incremental compile

**Goal:** Default compile processes only new leaves and fits them to existing branch structure. Full recompile is opt-in.

- [ ] Persist `last_compiled` timestamp (in config, manifest, or tree metadata)
- [ ] On `bo compile`: identify leaves where `collected > last_compiled`
- [ ] If no new leaves and no stale branches: report "nothing to compile", exit 0
- [ ] LLM prompt receives: existing branch structure (titles + summaries) + new leaf summaries
- [ ] LLM outputs: assignments of new leaves to existing branches, and/or new branch definitions
- [ ] Existing branch prose is untouched (new leaves are appended/integrated, not full rewrite)
- [ ] `--all` flag: ignore `last_compiled`, recompile full corpus (current behavior)
- [ ] Detect deleted leaves: if a leaf referenced by a branch no longer exists, flag branch as stale
- [ ] `--rebuild-stale` flag: recompile only stale branches from their remaining leaves
- [ ] Update `last_compiled` on successful compile
- [ ] `compile_model` config key threaded through (may need different model from query)
- [ ] Unit tests: incremental selection logic, stale detection
- [ ] Integration tests: collect → compile → collect more → compile again (only new processed)

**Done when:** A 50+ leaf tree with 3 new additions compiles in seconds (touching only new leaves), and a manually deleted leaf causes its branch to be flagged stale.

---

### 4. Query no-answer hardening

**Goal:** When retrieval finds irrelevant content, query reports "no answer" rather than synthesizing from noise.

- [ ] Implement retrieval relevance floor (minimum match quality before invoking synthesis)
- [ ] Generic-term filtering: very common terms that match many unrelated leaves don't trigger synthesis
- [ ] If synthesis runs but produces zero valid citations: treat as not-answered (extend existing zero-citation logic)
- [ ] Human output: clear "no relevant sources found" messaging
- [ ] JSON output: `insufficient_sources` or `low_relevance` error code
- [ ] Unit tests: below-threshold retrieval → no synthesis call
- [ ] Integration test: query unrelated topic against a themed tree → explicit not-answered

**Done when:** `bo query "quantum chromodynamics"` against a Rust programming tree exits with clear "no answer" rather than attempting synthesis against string-matched noise.

## Stretch scope

### 5. Snapshot manifest

**Goal:** Capture tree state before compile for safety and diff-driven decisions.

- [ ] Before compile: record file paths, content hashes, frontmatter summaries, index state
- [ ] Store snapshot under `~/.bo/snapshots/` or similar
- [ ] Diff current state against previous snapshot
- [ ] Feed diff into incremental compile: detect deletions, manual edits, new files
- [ ] Unit tests: snapshot creation, diff detection

**Done when:** Compile creates a snapshot before running, and a subsequent compile can diff against it to detect changes.

---

### 6. Compile dry-run

**Goal:** Users can preview what compile would do without writing.

- [ ] `bo compile --dry-run` runs full pipeline (read → LLM call → validate) but prints proposed writes
- [ ] Show: new branches to create, existing branches to update, frontmatter changes
- [ ] Optionally cache the LLM response for subsequent `bo compile --apply` (or just re-run)
- [ ] `--json` output for dry-run results

**Done when:** `bo compile --dry-run` shows proposed changes without modifying the tree.

---

### 7. `bo raze` auth separation

**Goal:** Razing a tree doesn't destroy API key configuration.

- [ ] Stop deleting `auth.json` by default
- [ ] Add `--include-auth` flag for full wipe if explicitly desired
- [ ] Update raze confirmation message to reflect what will be deleted
- [ ] Unit tests: raze without flag preserves auth, with flag deletes auth

**Done when:** `bo raze` followed by `bo seed ~/new-tree` doesn't require re-running `bo config auth`.

## Out of scope (future releases)

- Provider modules / OpenRouter / Ollama (→ v0.1.0)
- Query V2 agent loop (→ post v0.1.0, per ADR-003)
- Async concurrent collection (→ v0.1.0 or later)
- New collection adapters (PDF, RSS, local markdown)
- Multitree support
- `bo bloom` / autonomous research
- Hosted/SaaS infrastructure

## Dependencies and sequencing

```
bo status ──────────────┐
  (last_compiled ts)    │
                        ├──→ incremental compile (needs timestamp tracking)
                        │
validation gate ────────┘──→ incremental compile (gate must exist before new compile logic)
                        │
                        ├──→ snapshot manifest (stretch, uses same state tracking)

query hardening ── independent, can ship anytime
raze auth ── independent, can ship anytime
```

## Success criteria

- A tree with 50+ leaves can `bo compile` in seconds (only processing new leaves), not minutes.
- `bo status` tells you at a glance what's new and uncompiled.
- Compile never writes malformed branches (validation gate catches bad LLM output).
- Query never presents hallucinated answers as grounded (zero-citation = explicit "no answer").

## Release mechanics

2. Update CHANGELOG.md.

4. Merge, tag `v0.0.2`, push tag → release workflow runs.
